💻 Nota sobre el Frontend y los Contenedores

A lo largo de este libro, todos los servicios backend corren en contenedores Docker: PostgreSQL, Item Service, API Gateway y Jenkins. Sin embargo, el frontend se trabaja de forma local usando Live Server de VS Code (puerto 5500), no en un contenedor.

¿Por qué? Es una decisión pedagógica deliberada. Live Server ofrece hot reload automático: cuando editas HTML, CSS o JavaScript, el navegador se actualiza instantáneamente sin necesidad de reconstruir contenedores. Esto acelera enormemente el desarrollo y te permite concentrarte en aprender la arquitectura de microservicios sin fricciones innecesarias.

¿Quieres todo 100% contenedorizado? Si prefieres una arquitectura completamente consistente donde el frontend también corra en un contenedor (como se hace en producción), el Módulo 10, Sección 10.5: Frontend en Contenedor con Nginx (Opcional) te enseña paso a paso cómo lograrlo. Allí aprenderás a:

  • Crear un Dockerfile para el frontend usando Nginx Alpine
  • Configurar Nginx como reverse proxy para las peticiones a la API
  • Agregar el servicio frontend al docker-compose.yml
  • Entender las ventajas y desventajas frente a Live Server

Esta sección es completamente opcional. Puedes seguir todo el libro con Live Server y, si más adelante necesitas contenerizar el frontend para producción, volver al Módulo 10 cuando lo necesites.

⚙ Nota sobre Jenkins y el Flujo de Desarrollo

Durante el desarrollo normal de este libro, NO necesitas Jenkins. El flujo de trabajo principal es simple y automático:

Flujo Principal de Desarrollo (sin Jenkins):

  1. Editas código Java en VS Code (ej: ItemController.java)
  2. Guardas el archivo (Ctrl+S)
  3. Spring Boot DevTools detecta el cambio automáticamente
  4. El contenedor se recarga en ~2 segundos (hot reload)
  5. Pruebas los cambios en el navegador o Postman

Este flujo es automático, rápido y funciona perfectamente para todo el libro.

¿Qué es Jenkins entonces? Jenkins es un servidor de CI/CD (Integración Continua/Despliegue Continuo) que automatiza tareas como: compilar el código, ejecutar tests, construir imágenes Docker y desplegar actualizaciones. Es útil para:

  • Automatizar el proceso de build y test cuando haces push a Git
  • Construir imágenes Docker automáticamente
  • Desplegar nuevas versiones sin intervención manual
  • Entornos de producción donde necesitas pipelines de despliegue

¿Cuándo usar Jenkins? Si quieres aprender sobre CI/CD y automatización avanzada, el Módulo 10, Sección 10.4: CI/CD Local con Jenkins te enseña paso a paso cómo configurarlo. Pero no es necesario para seguir el resto del libro.

Recomendación: Sigue el libro normalmente con el flujo de hot reload automático. Cuando termines y quieras aprender sobre automatización CI/CD, vuelve al Módulo 10.

🔬 Puntos Clave del Flujo
  • Separación de responsabilidades: Cada componente hace UNA cosa bien. El Gateway solo enruta, el Service solo aplica lógica de negocio, el Repository solo habla con la BD.
  • Conversión de datos: JSON → DTO → Entity → SQL → Entity → DTO → JSON. Cada conversión tiene un propósito específico.
  • Red interna Docker: El Gateway y el Item Service se comunican por nombres (item-service:8081), no por IPs. Docker DNS resuelve automáticamente.
  • Transaccionalidad: @Transactional garantiza que si algo falla a mitad del proceso, se hace rollback y la BD no queda en estado inconsistente.
  • Seguridad CORS: El navegador bloquearía la petición si el Gateway no agregara los headers CORS. Por eso el Gateway los maneja centralizadamente.

⚠️ Por Qué el Item Service NO Está Expuesto al Host?

En el docker-compose.yml, el Item Service tiene ports: ["8081:8081"], pero esto es solo para debugging. En producción, solo el Gateway debería estar expuesto.

Esto se llama principio de mínimo privilegio: el Item Service no necesita ser accesible desde fuera de la red Docker. Solo el Gateway necesita exponer puertos al mundo exterior.

Beneficios:

  • Seguridad: Menos superficie de ataque. Solo el Gateway está expuesto.
  • Control: El Gateway puede aplicar autenticación, rate limiting, logging.
  • Flexibilidad: Puedes cambiar el Item Service sin afectar los clientes externos.
🔬 ¿Por qué un API Gateway?
  • Enrutamiento: Dirige peticiones al microservicio correcto. /api/items/** va al Item Service.
  • CORS Centralizado: Maneja Cross-Origin en un solo lugar, no en cada microservicio.
  • Seguridad: Puede validar tokens JWT, rate limiting antes de llegar a los servicios.
  • Aislamiento: Los microservicios internos no necesitan estar expuestos al exterior.

0.2 Las 4 Capas del Microservicio (Clean Architecture)

Clean Architecture organiza el codigo en capas concentricas. Cada capa solo depende de la capa inmediatamente inferior, nunca de capas superiores. Esto garantiza que la logica de negocio permanezca aislada de detalles tecnicos como frameworks o bases de datos.

CAPA 4 Controller

Recibe HTTP y devuelve JSON. Convierte DTOs a entidades. Es la unica capa que conoce HTTP.

↓↓ Delega ↓↓
CAPA 3 Service

Logica de negocio, validaciones, transformaciones DTO-Entity. No sabe de HTTP ni de SQL.

↓↓ Delega ↓↓
CAPA 2 Repository

Interfaz con la BD. Spring Data JPA genera SQL automaticamente a partir de nombres de metodos.

↓↓ Mapea ↓↓
CAPA 1 Entity

Representacion Java de la tabla de la base de datos. Define la estructura de los datos.

🔬 ¿Por qué separar en capas?

Imagina que quieres cambiar PostgreSQL por MongoDB. Si todo el codigo estuviera mezclado, tendrias que modificar cientos de lineas. Con Clean Architecture, solo cambias la capa Repository y el Entity. Las capas Service y Controller permanecen intactas porque no conocen los detalles de la base de datos.

Otro beneficio: cada capa puede testearse de forma aislada. Puedes probar la logica de negocio (Service) sin necesidad de tener una base de datos corriendo.

Modulo 2

Git: Control de Versiones Profesional

Antes de darle vida a nuestro input, prepararemos nuestra máquina del tiempo. Sin Git, no hay ingeniería de software segura.

2.1 ¿Qué es Git y Por Qué es Obligatorio?

Git es un sistema de control de versiones distribuido. Cada vez que haces un "commit", tomas una fotografia del estado actual de tu proyecto. Si algo se rompe, puedes volver a cualquier fotografia anterior. Es como tener infinitos "Ctrl+Z" que sobreviven a reinicios, formateos y desastres.

Git es distribuido porque cada desarrollador tiene una copia completa del historial en su maquina. No dependes de un servidor central para trabajar (aunque GitHub/GitLab sirven para sincronizar).

⚠️ Git NO es GitHub

Son herramientas diferentes que trabajan juntas:

  • Git: Software que corre en tu maquina. Gestiona el historial localmente. Puedes usar Git sin GitHub.
  • GitHub/GitLab: Servicios en la nube que almacenan repositorios Git remotamente. Sirven para colaboracion, backup y CI/CD. Son opcionales para desarrollo individual.
💻 CONSOLA - Instalar y Configurar Git
# Verificar instalacion (debe mostrar version 2.40+) git --version # Configurar identidad (obligatorio para commits) # Git necesita saber quien eres para registrar los cambios git config --global user.name "Tu Nombre" git config --global user.email "tu@email.com" # Verificar configuracion git config --global --list # Establecer rama principal como "main" (estandar moderno) git config --global init.defaultBranch main

La bandera --global aplica la configuracion a todos tus repositorios. Sin ella, solo aplica al repositorio actual.

2.2 Inicializar Repositorio y .gitignore

Un repositorio es la carpeta donde Git rastrea cambios. Inicializarlo crea una carpeta oculta .git/ que contiene todo el historial.

💻 CONSOLA - git init
# Ir a la carpeta del proyecto cd estudio-docker-microserv # Inicializar repositorio (crea carpeta .git/ oculta) git init # Ver estado (todo esta "untracked" = sin rastrear) git status # Crear .gitignore ANTES del primer commit # (ver codigo abajo) # Anadir todos los archivos al "staging area" git add . # Primer commit (fotografia inicial del proyecto) git commit -m "Initial commit: estructura base del proyecto"

Conceptos clave:

  • git add . marca todos los archivos para el proximo commit.git commit toma la fotografia. El mensaje (-m) describe que cambios contiene.
  • git status muestra que archivos han cambiado desde el ultimo commit.
⚠️ IMPORTANTE: No clones el repositorio del libro

El comando git clone que aparece en muchos tutoriales apunta al repositorio de ESTE LIBRO en GitHub. Si lo usas, descargarás el código del libro, NO crearás tu propio repositorio.

Lo que DEBES hacer:

  1. Crea tu PROPIO repositorio vacío en GitHub (sin README, sin .gitignore, sin licencia)
  2. Copia la URL de TU repositorio (ej: https://github.com/TU_USUARIO/TU_REPO.git)
  3. En tu carpeta local, en vez de git clone, haz:
    # Agregar tu repositorio remoto git remote add origin https://github.com/TU_USUARIO/TU_REPO.git # Subir tu codigo por primera vez git push -u origin main

Por que es importante: Tu repositorio es donde guardarás TU código, tus commits, tu historia. El repositorio del libro es solo para consulta. Si haces git clone del libro, no podrás subir tus cambios (no tienes permisos) y mezclarás el código del libro con el tuyo.

💻 CONSOLA - Conectar TU repositorio
# Despues de git init y tu primer commit local git remote add origin https://github.com/TU_USUARIO/TU_REPO.git # Verificar que se agrego correctamente git remote -v # Subir por primera vez (crea la rama main en remoto) git push -u origin main
# ═══════════════════════════════════════════════════════════════ # ARCHIVOS QUE GIT DEBE IGNORAR # ═══════════════════════════════════════════════════════════════ # Variables de entorno con credenciales (NUNCA subir secretos) .env .env.local .env.*.local # IDE y editores (configuraciones personales) .vscode/ .idea/ *.iml # Archivos de sistema .DS_Store Thumbs.db # Java compilado (Maven lo compila dentro del contenedor) target/ *.class *.jar *.war # Logs *.log logs/ # Node (si hubiera frontend Node) node_modules/ # Docker (no incluir en build context) .docker/
⚠️ ¿Por qué ignorar .env?

El archivo .env contiene credenciales (usuario, contraseña de PostgreSQL). Si lo subes a Git (especialmente a GitHub publico), cualquiera podra ver tus secretos. Por eso .gitignore lo excluye. En su lugar, sube un archivo .env.example con valores de ejemplo (sin credenciales reales) para que otros desarrolladores sepan que variables necesitan.

2.3 Commits y Ramas: Flujo de Trabajo

Las ramas (branches) te permiten trabajar en funcionalidades aisladas sin afectar la rama principal. Cuando terminas, "mergeas" (fusionas) la rama de vuelta a main.

Flujo de Ramas

main:       A---B-----------E---F (merge)
                 \         /
feature:          C---D---

A, B = commits iniciales. C, D = trabajo en feature. E = merge. F = siguiente commit en main.

💻 CONSOLA - Flujo de Trabajo con Ramas
# Renombrar rama principal a main (estandar moderno) git branch -M main # Crear rama para nueva funcionalidad y cambiar a ella # "-b" crea la rama, "checkout" cambia a ella git checkout -b feature/item-service # Trabajar, hacer cambios en archivos... # Cuando termines una unidad logica de trabajo: git add . git commit -m "Add Item entity and repository" # Otro cambio (por ejemplo, agregar Controller): git add . git commit -m "Add ItemController with 6 endpoints" # Ver historial de commits (grafico de ramas) git log --oneline --graph # Volver a rama main git checkout main # Fusionar la rama feature a main git merge feature/item-service # Ver todas las ramas (la rama feature sigue existiendo) git branch -a # Eliminar rama feature (ya no la necesitamos) git branch -d feature/item-service
🔬 Que significa "/" en feature/item-service?

La barra / en los nombres de ramas es una convención de nomenclatura que organiza las ramas por categorías, similar a como las carpetas organizan archivos en tu sistema operativo.

Estructura: categoria/nombre-especifico

PrefijoPropósitoEjemplo
feature/Nueva funcionalidadfeature/item-service, feature/login
bugfix/Corregir bug en desarrollobugfix/fix-login-error
hotfix/Corregir bug urgente en producciónhotfix/security-patch
release/Preparar versión de lanzamientorelease/v1.2.0

Importante: Git NO crea carpetas reales. feature/item-service es simplemente el nombre completo de la rama. La barra es solo un carácter más en el nombre, pero ayuda visualmente a agrupar ramas relacionadas.

Beneficios:

  • Organización: Cuando ejecutas git branch -a, ves todas las ramas de una categoría juntas
  • Claridad: Sabes inmediatamente el propósito de la rama
  • Convención de equipo: Todos siguen el mismo patrón de nomenclatura

Puedes usar cualquier nombre sin barra (ej: item-service), pero usar prefijos es una práctica estándar en desarrollo profesional.

⚠️ Buenas Practicas de Commits
  • Commits atomicos: Cada commit = un cambio logico unico. No mezcles "añadir entity" con "configurar docker".
  • Mensajes en imperativo: "Add feature", "Fix bug", "Update config". No "Added", "Fixed", "Updated".
  • Nunca subas .env: Contiene credenciales. Siempre en .gitignore.
  • Frecuencia: Commit despues de completar una tarea pequena. Mejor muchos commits pequeños que uno gigante.

El cerebro detrás del input. Construiremos las 4 capas que procesarán, validarán y guardarán el texto que el usuario escriba.

3.1 pom.xml

🔬 ¿Por qué estamos creando este archivo?

¿Qué es pom.xml? POM significa Project Object Model (Modelo de Objeto del Proyecto). Es el archivo de configuracion central de cualquier proyecto Maven. Piensa en ello como el "DNI" de tu aplicacion Java: le dice a Maven que proyecto es, que dependencias necesita, como compilarlo y como empaquetarlo.

Por que es critico? Sin pom.xml, Maven no sabe:

  • Que version de Java usar
  • Que librerias descargar (Spring Boot, PostgreSQL, etc.)
  • Cómo compilar el codigo fuente
  • Cómo empaquetar la aplicacion en un archivo .jar ejecutable

¿Qué pasaría sin este archivo? Sin pom.xml, no puedes compilar tu microservicio. Maven no tendria idea de que librerias descargar ni como construir tu proyecto. Es tan esencial como el motor de un auto.

Analogia: Si tu microservicio fuera un recipe de cocina, el pom.xml seria la lista de ingredientes + las instrucciones de preparacion. Sin la lista, no sabes que comprar; sin las instrucciones, no sabes como cocinar.

Crea item-service/pom.xml:

<?xml version="1.0" encoding="UTF-8"?> <project xmlns="http://apache.org" xmlns:xsi="http://w3.org" xsi:schemaLocation="http://apache.org https://apache.org"> <modelVersion>4.0.0</modelVersion> <parent> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-parent</artifactId> <version>3.4.5</version> <relativePath/> </parent> <groupId>com.singleinput</groupId> <artifactId>item-service</artifactId> <version>0.0.1-SNAPSHOT</version> <properties> <java.version>21</java.version> </properties> <dependencies> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-data-jpa</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-validation</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-devtools</artifactId> <scope>runtime</scope> <optional>true</optional> </dependency> <dependency> <groupId>org.postgresql</groupId> <artifactId>postgresql</artifactId> <scope>runtime</scope> </dependency> <dependency> <groupId>org.projectlombok</groupId> <artifactId>lombok</artifactId> <optional>true</optional> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-test</artifactId> <scope>test</scope> </dependency> </dependencies> <build> <plugins> <plugin> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-maven-plugin</artifactId> <configuration> <excludes> <exclude> <groupId>org.projectlombok</groupId> <artifactId>lombok</artifactId> </exclude> </excludes> </configuration> </plugin> </plugins> </build> </project>

3.2 application.yml

🔬 ¿Por qué estamos creando este archivo?

¿Qué es application.yml? Es el archivo de configuracion principal de Spring Boot. Aqui defines como tu aplicacion se conecta a la base de datos, en que puerto escucha, que nivel de logging usar, etc. Es el "panel de control" de tu microservicio.

Por que YAML y no properties? YAML usa indentacion (como Python) en vez de clave=valor. Es mas legible para configuraciones anidadas. Spring Boot acepta ambos formatos, pero YAML es el estandar en proyectos profesionales.

¿Qué pasaría sin este archivo? Spring Boot usaria valores por defecto: puerto 8080, H2 en memoria (no PostgreSQL), sin mostrar SQL. Tu microservicio arrancaria pero no se conectaria a la base de datos correcta.

3.3 Entity + DTO

🔬 ¿Por qué estamos creando estos archivos?

Item.java (Entity - CAPA 1): Es la representacion Java de la tabla items en PostgreSQL. Cada campo de la clase se mapea a una columna de la tabla. Sin Entity, JPA/Hibernate no sabe como guardar objetos Java en la base de datos.

ItemDTO.java (Data Transfer Object): Es el "mensajero" que transporta datos entre capas. Por que no usar Entity directamente? Porque el DTO permite:

  • Validacion: @NotBlank, @Size verifican que los datos sean correctos ANTES de llegar a la BD
  • Seguridad: Puedes ocultar campos sensibles (como contraseñas) que no deberian exponerse al cliente
  • Desacoplamiento: Si cambias la estructura de la tabla, el DTO puede permanecer igual

¿Qué pasaría sin estos archivos? Sin Entity, no hay tabla en la BD. Sin DTO, no hay forma de validar datos de entrada ni controlar que campos se exponen al cliente.

3.4 Repository

🔬 ¿Por qué estamos creando este archivo?

¿Qué es ItemRepository? Es una interfaz que Spring Data JPA implementa automaticamente. Tu solo declaras QUE operaciones necesitas (find, save, delete), y Spring genera el codigo SQL por ti.

La magia de los nombres de metodos: Spring Data JPA "lee" el nombre del metodo y genera el SQL automaticamente. findByTextoContainingIgnoreCase se traduce a: SELECT * FROM items WHERE LOWER(texto) LIKE LOWER('%texto%'). No necesitas escribir SQL.

¿Qué pasaría sin este archivo? Tendrias que escribir SQL manualmente con JDBC (Connection, PreparedStatement, ResultSet). Son cientos de lineas propensas a errores. El Repository elimina todo ese boilerplate.

item-service/src/main/java/com/singleinput/persistence/ItemRepository.java:

// Paquete "persistence": capa de acceso a datos (CAPA 2) package com.singleinput.persistence; // Importamos la Entity que este repository maneja import com.singleinput.domain.Item; // JpaRepository: interfaz generica de Spring Data que incluye // metodos CRUD estandar: save(), findAll(), findById(), delete(), etc. import org.springframework.data.jpa.repository.JpaRepository; import java.util.List; // Interfaz (no clase) que extiende JpaRepository // <Item, Long> = maneja Entity "Item" con clave primaria tipo Long // Spring crea automaticamente una implementacion de esta interfaz // en tiempo de ejecucion (un "proxy" que ejecuta el SQL real) public interface ItemRepository extends JpaRepository<Item, Long> { // Metodo personalizado: Spring Data JPA "lee" el nombre // findBy = SELECT ... WHERE // Texto = campo de la Entity // Containing = LIKE '%valor%' (busqueda parcial) // IgnoreCase = sin distinguir mayusculas/minusculas // SQL generado: SELECT * FROM items WHERE LOWER(texto) LIKE LOWER('%valor%') List<Item> findByTextoContainingIgnoreCase(String texto); }

3.5 Service

🔬 ¿Por qué estamos creando estos archivos?

¿Qué es ItemService? Es la CAPA 3 (logica de negocio). Aqui vive toda la inteligencia de la aplicacion: validaciones, transformaciones entre Entity y DTO, y reglas de negocio. El Controller delega todo al Service.

¿Por qué una interface + implementacion? El patron Interface + Impl permite:

  • Testeabilidad: En tests unitarios, puedes "mockear" (simular) el Service sin necesidad de base de datos
  • Flexibilidad: Puedes cambiar la implementacion (ej: de JPA a MongoDB) sin tocar el Controller
  • Contrato claro: La interface define QUE hace el service; la implementacion define COMO

¿Qué pasaría sin esta capa? El Controller tendria que hacer TODO: recibir HTTP, validar datos, convertir DTOs, hablar con la BD. Seria un "Dios objeto" imposible de mantener y testear.

service/ItemService.java (interface):

// Paquete "service": capa de logica de negocio (CAPA 3) package com.singleinput.service; import com.singleinput.dto.ItemDTO; import java.util.List; // Interface: define el CONTRATO que debe cumplir la implementacion // El Controller solo conoce esta interface, no la implementacion // Esto se llama "Inversion de Dependencia" (principio SOLID) public interface ItemService { // Retorna todos los items como DTOs (no como Entities) List<ItemDTO> findAll(); // Busca un item por su ID. Lanza excepcion si no existe ItemDTO findById(Long id); // Crea un nuevo item a partir de un DTO. Retorna el DTO con ID generado ItemDTO create(ItemDTO dto); // Actualiza un item existente. Lanza excepcion si no existe ItemDTO update(Long id, ItemDTO dto); // Elimina un item por ID. Lanza excepcion si no existe void delete(Long id); // Busca items cuyo texto contenga la cadena indicada (case-insensitive) List<ItemDTO> search(String texto); }

Es "final" porque se inyecta // una sola vez en el constructor (inmutabilidad) private final ItemRepository repo; // Constructor: Spring inyecta automaticamente el ItemRepository // Esto se llama "Inyeccion por Constructor" (mejor practica) // Spring busca un bean de tipo ItemRepository y lo pasa aqui public ItemServiceImpl(ItemRepository repo) { this.repo = repo; } // @Override: Indica que estamos implementando un metodo de la interface // @Transactional(readOnly = true): Optimizacion de rendimiento // Indica a Hibernate que NO necesita preparar rollback // Solo lectura = mas rapido que lectura+escritura @Override @Transactional(readOnly = true) public List<ItemDTO> findAll() { // repo.findAll() retorna List<Item> (entities) // .stream().map(this::toDTO) convierte cada Entity a DTO // .collect() convierte el stream de vuelta a List<ItemDTO> return repo.findAll().stream().map(this::toDTO).collect(Collectors.toList()); } @Override @Transactional(readOnly = true) public ItemDTO findById(Long id) { // repo.findById() retorna Optional<Item> // .orElseThrow() lanza excepcion si el Optional esta vacio // Esto reemplaza el clasico "if (item == null) throw..." return toDTO(repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id))); } // @Transactional (sin readOnly): Este metodo MODIFICA datos // Si algo falla despues de save(), se hace rollback @Override @Transactional public ItemDTO create(ItemDTO dto) { // 1. Crear Entity vacia // 2. Copiar datos del DTO al Entity // 3. repo.save() INSERTA en la BD y retorna el Entity con ID generado // 4. Convertir el Entity resultante de vuelta a DTO Item e = new Item(); e.setTexto(dto.getTexto()); return toDTO(repo.save(e)); } @Override @Transactional public ItemDTO update(Long id, ItemDTO dto) { // 1. Buscar el item existente (o lanzar excepcion si no existe) Item ex = repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id)); // 2. Modificar los campos del entity existente ex.setTexto(dto.getTexto()); // 3. repo.save() hace UPDATE porque el entity ya tiene ID return toDTO(repo.save(ex)); } @Override @Transactional public void delete(Long id) { // Verificar que el item existe antes de intentar borrarlo // repo.existsById() genera: SELECT COUNT(*) > 0 FROM items WHERE id = ? if (!repo.existsById(id)) throw new ItemNotFoundException(id); // Eliminar el item. 3.6 Controller

🔬 ¿Por qué estamos creando este archivo?

¿Qué es ItemController? Es la CAPA 4 (la mas externa). Es el "recepcionista" de tu microservicio: recibe peticiones HTTP del mundo exterior, las delega al Service, y devuelve respuestas HTTP con datos en formato JSON.

¿Qué pasaría sin este archivo? No habria forma de que el frontend, Postman o el API Gateway se comuniquen con tu microservicio. Los endpoints REST no existirian. Es la unica capa que "habla HTTP".

Regla de oro: El Controller es "tonto". No tiene logica de negocio. Solo recibe, delega y responde. Toda la inteligencia esta en el Service.

3.7 Exception Handler

🔬 ¿Por qué estamos creando estos archivos?

ItemNotFoundException: Es una excepcion personalizada que se lanza cuando un item no existe. Extiende RuntimeException (excepcion no verificada). Sin ella, tendrias que retornar nulls o codigos de error genericos.

GlobalExceptionHandler: Es un "paracaidas de seguridad" global. Cuando cualquier excepcion ocurre en cualquier Controller, este handler la captura y retorna un JSON de error estructurado (en vez de un stack trace feo de 500 lineas).

¿Qué pasaría sin estos archivos? Cuando un item no existe, Spring retornaria un error 500 con un mensaje generico tipo "Internal Server Error". El frontend no sabria distinguir entre "item no encontrado" (404) y "error del servidor" (500). Con el handler global, cada error tiene su propio status code y mensaje claro.

3.8 Application

🔬 ¿Por qué estamos creando este archivo?

¿Qué es ItemServiceApplication? Es el punto de entrada de tu microservicio (el "main" de Java). Cuando ejecutas mvn spring-boot:run, Java busca este metodo main() para arrancar todo.

La magia de @SpringBootApplication: Esta unica anotacion combina tres cosas:

  • @Configuration: "Esta clase puede definir beans"
  • @EnableAutoConfiguration: "Spring, configura automaticamente todo lo que encuentres en el classpath" (si ve PostgreSQL en pom.xml, configura la conexion; si ve spring-boot-starter-web, arranca Tomcat)
  • @ComponentScan: "Busca todas las clases con @Service, @Repository, @Controller en este paquete y subpaquetes, y registralas como beans"

¿Qué pasaría sin este archivo? Tu microservicio no arrancaria. No habria un punto de entrada para Spring Boot. Es tan esencial como el interruptor de encendido de un auto.

4.1 Contenedor vs Maquina Virtual

Un contenedor es un proceso del sistema operativo aislado. No es una maquina virtual completa (no tiene su propio kernel), pero tiene su propio sistema de archivos, tabla de procesos, red y espacio IPC. Comparte el kernel del host con otros contenedores.

Maquina Virtual (VM)

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|
| Guest OS | | Guest OS |  <- OS completo por VM
+----------+-+----------+
|     Hypervisor       |  <- Capa pesada
+----------------------+
|      Host OS         |
+----------------------+

Contenedor Docker

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|  <- Solo lo necesario
+----------+-+----------+
|    Docker Engine     |  <- Capa ligera
+----------------------+
|   Host OS Kernel     |  <- Compartido
+----------------------+

La consecuencia practica es brutal: un contenedor arranca en menos de 1 segundo (una VM tarda 30-60s), consume ~50MB de RAM (una VM necesita 1-2GB), y la imagen de PostgreSQL Alpine pesa solo ~80MB (una VM con Linux + PostgreSQL serian varios GB).

4.3 Las 3 Reglas de Oro de las IPs en Docker

Este es el concepto mas importante del libro. Si lo entiendes, todo lo demas sera natural. Si lo ignoras, pasaras horas debuggeando errores de conexion.

⚠️ REGLAS FUNDAMENTALES

Regla 1: "localhost" es egosta

Dentro de un contenedor, localhost = el mismo contenedor. Si Item Service dice localhost:5432, busca PostgreSQL dentro de SI MISMO (y falla). Nunca uses localhost entre contenedores.

Regla 2: Los contenedores se llaman por nombre

Las IPs (172.17.0.x) son efimeras. El nombre (crud-postgres) es estable. Docker DNS resuelve nombres a IPs. Por eso usamos host: crud-postgres en application.yml.

Regla 3: El Host ve contenedores por puertos mapeados

Desde tu navegador, accedes con localhost:PUERTO_HOST. El mapeo -p 8080:8080 crea un tunel entre tu maquina y el contenedor.

4.4 Volumenes: Persistencia de Datos

🔬 Que son los volumenes de Docker y por que existen?

El problema del almacenamiento efimero: Por defecto, todos los archivos creados DENTRO de un contenedor viven en su sistema de archivos interno. Cuando eliminas el contenedor (docker compose down), TODO se borra permanentemente. Esto es como escribir en una pizarra: todo lo que escribes desaparece cuando la borras.

Por que Docker hace esto? Los contenedores estan disenados para ser efimeros (temporales). Puedes destruirlos y recrearlos cuantas veces quieras. Esto es genial para aplicaciones sin estado (stateless), pero es un DESASTRE para bases de datos. Imagina que cada vez que reinicias tu servidor, se borran todos los usuarios, productos y pedidos de tu aplicacion.

La solucion - Volumenes: Un volumen es una zona de almacenamiento que vive FUERA del ciclo de vida del contenedor. Piensa en ello como un "disco duro externo" que conectas al contenedor. Puedes destruir el contenedor mil veces, pero el volumen (y sus datos) permanecen intactos.

Analogia del edificio de apartamentos: Si un contenedor es un apartamento amoblado, el volumen es una bodega externa. Si te mudas (destruyes el contenedor), tus cosas en la bodega (volumen) siguen ahi. Cuando te mudas a un nuevo apartamento (nuevo contenedor), puedes volver a conectar la misma bodega.

Por que NECESITAMOS un volumen para PostgreSQL en este libro?

⚠️ Escenario sin volumen

Imagina esta secuencia:

  1. Arrancas todo con docker compose up -d
  2. Creas items desde Postman: "Hola mundo", "Test 1", "Test 2"
  3. Los datos se guardan en PostgreSQL DENTRO del contenedor
  4. Apagas todo con docker compose down
  5. Al dia siguiente, arrancas de nuevo con docker compose up -d
  6. SIN VOLUMEN: Todos los items desaparecieron. La tabla "items" esta vacia.
  7. CON VOLUMEN: Todos los items siguen ahi. La tabla "items" tiene tus datos.

El volumen crud_pgdata es lo que separa el escenario 6 del 7. Sin el, cada reinicio seria como formatear la base de datos.

Paso a paso: Cómo se crea el volumen en docker-compose.yml

Paso 1: Declarar el volumen (seccion "volumes:")

# Al final de docker-compose.yml, fuera de "services:" volumes: # "crud_pgdata" es el nombre del volumen # Docker lo crea automaticamente la primera vez que arrancas crud_pgdata: driver: local

Esto le dice a Docker: "crea un volumen llamado crud_pgdata usando el driver local". No necesitas crearlo manualmente con docker volume create; Docker Compose lo hace por ti.

Paso 2: Montar el volumen en el servicio (seccion "services: postgres: volumes:")

postgres: volumes: # Formato: "NOMBRE_VOLUMEN:RUTA_DENTRO_DEL_CONTENDOR" - crud_pgdata:/var/lib/postgresql/data

Esto conecta el volumen crud_pgdata con la ruta /var/lib/postgresql/data dentro del contenedor. PostgreSQL guarda TODOS sus datos en esa ruta. Ahora, en vez de escribir en el filesystem efimero del contenedor, escribe en el volumen persistente.

Que pasa cuando ejecutas docker compose up con volumenes?

1

Docker Compose lee docker-compose.yml

Identifica los volumenes declarados: crud_pgdata, maven_cache, gateway_maven_cache

2

Verifica si los volumenes ya existen

Si el volumen NO existe (primera vez), Docker lo crea vacio en /var/lib/docker/volumes/. Si YA existe, lo reutiliza con todos sus datos intactos.

3

Arranca el contenedor PostgreSQL

PostgreSQL detecta el volumen montado en /var/lib/postgresql/data. Si esta vacio, inicializa la base de datos. Si ya tiene datos, los usa directamente (no reinicia nada).

4

Tu aplicacion escribe datos

Cuando creas un item, PostgreSQL escribe en /var/lib/postgresql/data/base/... que en realidad es el volumen crud_pgdata en tu disco.

5

Ejecutas docker compose down

El contenedor se destruye, pero el volumen crud_pgdata permanece en tu disco con todos los datos. Cuando vuelvas a arrancar, todo estara ahi.

5.1 Dockerfile del Item Service

🔬 ¿Por qué estamos creando este archivo?

¿Qué es un Dockerfile? Es la "receta de cocina" que Docker usa para construir una imagen. Una imagen es una plantilla inmutable que contiene TODO lo necesario para ejecutar tu aplicacion: Java, Maven, tu codigo, tus dependencias, configuraciones. El Dockerfile le dice a Docker paso a paso como construir esa plantilla.

Por que lo necesitamos? Sin Dockerfile, Docker no sabe cómo crear la imagen de tu microservicio. No puede adivinar que version de Java usar, que dependencias descargar, o cómo arrancar tu aplicacion. El Dockerfile es el contrato entre tu codigo y el contenedor.

Analogia: Si la imagen Docker es un pastel, el Dockerfile es la receta. Sin receta, no sabes que ingredientes comprar ni en que orden mezclarlos. Cada instruccion del Dockerfile (FROM, RUN, COPY, CMD) es un paso de la receta.

Un Dockerfile es la "receta" que Docker usa para construir una imagen. Cada instruccion (FROM, RUN, COPY, etc.) crea una "capa" en la imagen. Docker cachea estas capas: si no cambias una instruccion, no re-ejecuta esa capa.

🔬 ¿Por qué un Dockerfile separado para el Gateway?

Cada microservicio tiene su propio Dockerfile porque son aplicaciones independientes con dependencias diferentes. El Item Service necesita JPA, PostgreSQL y Validation. 5.3 Archivo .dockerignore

🔬 ¿Por qué estamos creando este archivo?

¿Qué es .dockerignore? Es el "guardian" del build context. Cuando ejecutas docker compose build, Docker envia TODOS los archivos de la carpeta del proyecto al daemon de Docker (el "contexto de build"). Esto incluye archivos que NO necesitas: la carpeta .git (que puede pesar cientos de MB), node_modules, archivos temporales, tu archivo .env con credenciales, etc.

¿Qué pasaría sin este archivo? Docker enviaria TODOS los archivos al daemon, incluyendo:

  • .git/ = cientos de MB innecesarios que ralentizan el build
  • .env = credenciales que podrian quedar accidentalmente en la imagen
  • target/ = archivos compilados que Maven recompilara dentro del contenedor
  • .vscode/ = configuracion personal de tu editor

Analogia: Si Docker Compose es el director de orquesta, .dockerignore es la lista de "no incluir" para el escenario. Le dice al stage manager: "no subas estos objetos al escenario porque solo estorban".

Cuando ejecutas docker compose build, Docker envia todo el contenido de la carpeta actual (el "build context") al daemon. Esto incluye archivos que no necesitas: .git, node_modules, archivos temporales, etc. Para evitar esto, crea un archivo .dockerignore en la raiz del proyecto.

Modulo 6

API Gateway con Spring Cloud Gateway

La puerta de seguridad de nuestra aplicación. El input nunca hablará directamente con el backend, siempre pasará por esta aduana.

6.1 ¿Qué es un API Gateway?

Un API Gateway es un servidor que actua como puerta de entrada unica a todos los microservicios. El cliente (navegador) no sabe cuantos microservicios existen ni en que puertos corren; solo conoce la direccion del Gateway.

ⓘ Responsabilidades del Gateway
Enrutamiento: Redirige /api/items/** al microservicio item-service:8081.
CORS: Maneja los headers de Cross-Origin para permitir peticiones desde el frontend.
Seguridad: Punto centralizado para autenticacion, rate limiting, logging.
WebFlux: Spring Cloud Gateway usa programacion reactiva (NO MVC), lo que lo hace muy eficiente.

6.2 pom.xml del Gateway

🔬 ¿Por qué estamos creando este archivo?

¿Qué es este pom.xml? Es el archivo de configuracion Maven del API Gateway. Similar al pom.xml del Item Service, pero con dependencias diferentes: en vez de JPA y PostgreSQL, necesita Spring Cloud Gateway (que usa WebFlux reactivo en vez de MVC).

Diferencia clave con el Item Service: El Gateway usa spring-cloud-starter-gateway en vez de spring-boot-starter-web. Esto es porque el Gateway es reactivo (WebFlux), no bloqueante (MVC). Puede manejar miles de conexiones simultaneas con menos recursos.

¿Qué pasaría sin este archivo? El Gateway no tendria dependencias. No sabria que es Spring Cloud Gateway, no tendria WebFlux, no podria enrutar peticiones. Es tan esencial como el pom.xml del Item Service.

6.3 Configuracion del Gateway

🔬 ¿Por qué estamos creando estos archivos?

application.yml del Gateway: Le dice al Gateway DOS cosas criticas: (1) A donde reenviar las peticiones (rutas), y (2) Desde que origenes aceptar peticiones (CORS). Sin este archivo, el Gateway no sabria que hacer con las peticiones que recibe.

ApiGatewayApplication.java: Es el punto de entrada del Gateway (igual que ItemServiceApplication.java). Sin el, el Gateway no arranca.

¿Qué pasaría sin estos archivos? Sin application.yml, el Gateway no tiene rutas: todas las peticiones retornarian 404. Sin ApiGatewayApplication.java, no hay punto de entrada para arrancar el Gateway.

    Modulo 9

    JWT: Autenticacion Opcional (Sin Base de Datos)

    Protegiendo nuestro input. Implementaremos tokens JWT para asegurar que solo usuarios autorizados puedan crear o modificar datos.

    9.1 ¿Qué es JWT?

    JWT (JSON Web Token) es un estandar abierto (RFC 7519) para crear tokens de autenticacion. Un JWT es una cadena de texto firmada criptograficamente que contiene informacion del usuario (claims). La clave esta en que el servidor puede verificar la firma sin necesidad de consultar una base de datos, lo que lo hace ideal para microservicios.

    Problema que resuelve JWT

    Sin JWT, cada microservicio necesitaria su propio sistema de autenticacion o consultar una BD compartida en cada peticion. Con JWT, el API Gateway valida el token una vez y los microservicios confian en esa validacion.

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  AUTENTICACION ESTATICA: Sesion Tradicional vs JWT                                      │
└─────────────────────────────────────────────────────────────────────────────────────────┘

SESION TRADICIONAL (con BD):
  Cliente ───(login)───> Servidor ───> [BD: guarda sesion] ───> sessionId
  Cliente ───(request + sessionId)───> Servidor ───> [BD: busca sesion] ───> OK/Error
  ❌ Cada microservicio necesita consultar la BD para validar la sesion
  ❌ Escalar horizontalmente es complejo (sesiones atadas a un servidor)

JWT (sin BD):
  Cliente ───(login)───> Servidor ───> [firma JWT] ───> token
  Cliente ───(request + JWT)───> Servidor ───> [verifica firma] ───> OK/Error
  ✅ El token CONTIENE los datos del usuario (firmados, no encriptados)
  ✅ Cualquier microservicio puede verificar sin BD (solo necesita la clave secreta)
  ✅ Escalado horizontal simple: cualquier instancia puede validar cualquier token

    Estructura de un JWT

    Un JWT tiene 3 partes separadas por puntos (.), cada una codificada en Base64:

    // EJEMPLO DE JWT (coloreado por partes):
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJhZG1pbiIsInJvbGUiOiJBRE1JTiJ9.SflKxwRJSMeKKF2QT4fwpM
  ▲── HEADER ──▲       ▲── PAYLOAD ──▲       ▲── SIGNATURE ──▲

┌───────────────────────────────────────────────────────────────────────────────────────┐
│  HEADER (cabecera)                                                                       │
│  Contiene el algoritmo de firma y el tipo de token.                                     │
│  Ejemplo decodificado: {"alg": "HS256", "typ": "JWT"}                                  │
│  HS256 = HMAC con SHA-256 (firma simetrica, misma clave para firmar y verificar)       │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  PAYLOAD (carga util)                                                                    │
│  Contiene los "claims": datos del usuario y metadatos del token.                       │
│  Ejemplo: {"sub": "admin", "role": "ADMIN", "iat": 1712345678, "exp": 1712349278}      │
│  Claims estandar: sub (subject), iat (issued at), exp (expiration), iss (issuer)       │
│  Claims personalizados: role, permissions, tenantId...                                 │
├───────────────────────────────────────────────────────────────────────────────────────┤
│  SIGNATURE (firma)                                                                     │
│  Firma criptografica que verifica que el token no fue alterado.                        │
│  Se genera con: HMACSHA256( base64UrlEncode(header) + "." + base64UrlEncode(payload),  │
│                    secret )                                                            │
│  Si alguien modifica el header o payload, la firma ya no coincide → token invalido.    │
└───────────────────────────────────────────────────────────────────────────────────────┘

    ¿Como funciona el flujo completo?

    ┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  FLUJO DE AUTENTICACION CON JWT                                                       │
└─────────────────────────────────────────────────────────────────────────────────────────┘

PASO 1: Login
  Cliente (Postman/Frontend)
     │
     │ POST /api/auth/login
     │ {"username": "admin", "password": "admin123"}
     ▼
  API GATEWAY ───> AuthController
     │
     ├─ 1. Valida credenciales contra lista hardcodeada (o BD)
     ├─ 2. Crea JWT con: username, role, iat (fecha creacion), exp (fecha expiracion)
     ├─ 3. Firma el JWT con la clave secreta (HMAC-SHA256)
     └─ 4. Responde: {"token": "eyJhbGciOiJIUzI1NiJ9..."}
     │
     ▼
  Cliente recibe el token y lo almacena (en memoria, localStorage, etc.)

PASO 2: Peticion a recurso protegido
  Cliente
     │
     │ GET /api/items/listar
     │ Headers: Authorization: Bearer <JWT>
     ▼
  API GATEWAY ───> JwtAuthFilter
     │
     ├─ 1. Extrae el token del header "Authorization: Bearer ..."
     ├─ 2. Verifica la firma con la clave secreta
     ├─ 3. Verifica que el token no haya expirado (claim "exp")
     ├─ 4. Extrae username y role del payload
     └─ 5. Si es valido → reenvia al microservicio
         Si es invalido → responde 401 Unauthorized
     │
     ▼
  Item Service ───> Procesa la peticion normalmente (confia en el Gateway)

¿Que pasa si alguien intenta modificar el token?
  Token original:   header.payload.firma_valida
  Token modificado: header.payload_modificado.firma_invalida
  
  Al verificar: HMACSHA256(header + payload_modificado, secret) != firma_invalida
  Resultado: 🔴 TOKEN RECHAZADO (401 Unauthorized)
  
  La firma solo puede generarse con la clave secreta. Sin ella, es imposible
  crear un token valido o modificar uno existente sin invalidarlo.

    Paso 3: JwtUtil - Generar y validar tokens

    Crea la clase JwtUtil.java en api-gateway/src/main/java/com/gateway/security/. Esta clase es el corazon de la autenticacion: genera tokens cuando el usuario hace login, y los valida en cada peticion protegida.

    🔬 Analisis de JwtUtil

    @Component: Marca la clase como un bean de Spring. Spring creara una unica instancia y la inyectara donde sea necesaria.

    @Value("${...}"): Inyecta valores desde application.yml. secret y expiration se leen automaticamente.

    getSigningKey(): Convierte la clave secreta (String) en una SecretKey de Java, que es lo que necesita la libreria JJWT para firmar/verificar.

    generateToken(): Construye el JWT: asigna subject (username), claims (role), fechas de creacion y expiracion, y lo firma con HMAC-SHA256.

    validateToken(): Parsea el token, verifica la firma y la expiracion. Si todo esta bien, devuelve el username. Si no, devuelve null (token invalido).

    Nota: Si la firma no coincide o el token expiro, JJWT lanza una excepcion. El catch la captura y devuelve null.

    // JwtUtil.java - Utilidad para generar y validar tokens JWT // Ubicacion: api-gateway/src/main/java/com/gateway/security/JwtUtil.java package com.gateway.security; import io.jsonwebtoken.*; import io.jsonwebtoken.security.Keys; import org.springframework.beans.factory.annotation.Value; import org.springframework.stereotype.Component; import javax.crypto.SecretKey; import java.nio.charset.StandardCharsets; import java.util.Date; // @Component: Spring crea un bean unico y lo inyecta donde se necesite @Component public class JwtUtil { // @Value inyecta los valores desde application.yml @Value("${security.jwt.secret}") private String secret; @Value("${security.jwt.expiration}") private long expiration; // Convierte el String secreto en una clave criptografica private SecretKey getSigningKey() { byte[] keyBytes = secret.getBytes(StandardCharsets.UTF_8); return Keys.hmacShaKeyFor(keyBytes); } // Genera un JWT con username, role, fecha de creacion y expiracion public String generateToken(String username, String role) { return Jwts.builder() .subject(username) .claim("role", role) .issuedAt(new Date()) .expiration(new Date(System.currentTimeMillis() + expiration)) .signWith(getSigningKey()) .compact(); } // Valida un JWT: verifica firma + expiracion. Devuelve username o null public String validateToken(String token) { try { Claims claims = Jwts.parser() .verifyWith(getSigningKey()) .build() .parseSignedClaims(token) .getPayload(); return claims.getSubject(); } catch (Exception e) { return null; } } }

    ⚠ Resumen y Buenas Practicas

    Lo que logramos: El Gateway ahora protege todos los endpoints de los microservicios. Sin un JWT valido, el filtro rechaza la peticion con 401 antes de que llegue al microservicio.

    Limites de este ejemplo: Usuarios hardcodeados en application.yml (en produccion se usa base de datos o LDAP), clave secreta en texto plano (en produccion se usa Vault o variables de entorno), sin refresh tokens (el token expira en 1 hora y el usuario debe hacer login de nuevo).

    Mejoras para produccion: Almacenar claves en ${JWT_SECRET} (variable de entorno), agregar HTTPS para que el token no viaje en texto plano, usar refresh tokens rotativos, y agregar rate limiting en el endpoint de login para evitar ataques de fuerza bruta.

    Modulo 10

    CI/CD: Integracion y Despliegue Continuo

    El toque final profesional: automatizar las pruebas y el despliegue de nuestra aplicación cada vez que mejoremos nuestro input.

    10.1 ¿Qué es CI/CD?

    CI (Integracion Continua): Cada vez que haces push a Git, se ejecutan automaticamente: build, tests, analisis de codigo. Si algo falla, te notifican inmediatamente.

    CD (Despliegue Continuo): Si el CI pasa, el codigo se despliega automaticamente a staging o produccion. Sin intervencion manual.

    ⓘ Beneficios de CI/CD
    Deteccion temprana de errores: Los tests se ejecutan en cada push.
    Despliegues rapidos: De horas/manuales a minutos/automaticos.
    Consistencia: El mismo proceso se ejecuta siempre, sin errores humanos.
    Confianza: Si los tests pasan, puedes desplegar con seguridad.

    10.2 GitHub Actions

    GitHub Actions es el sistema de CI/CD de GitHub. Defines "workflows" en archivos YAML dentro de .github/workflows/. Cada workflow se dispara automaticamente segun eventos (push, pull request, etc.).

    10.3 Pipeline para Item Service

    🔬 ¿Por qué estamos creando este archivo?

    ¿Qué es este archivo? Es un "workflow" de GitHub Actions: un archivo YAML que define una serie de pasos automatizados que se ejecutan cada vez que haces push a la rama main. Es tu "asistente de calidad" que verifica automaticamente que el codigo compila y que los tests pasan.

    Que problema resuelve? Sin CI/CD, tendrias que recordar manualmente: compilar el proyecto, ejecutar los tests, verificar que nada se rompio. Con este archivo, GitHub hace todo eso automaticamente en cada push. Si algo falla, te notifica inmediatamente.

    ¿Qué pasaría sin este archivo? No habria verificacion automatica. Podrias subir codigo roto a main sin darte cuenta. En un equipo real, esto causaria conflictos y bugs en produccion.

    10.4 CI/CD Local con Jenkins

    🔬 ¿Por qué estamos creando esto?

    ¿Qué es Jenkins? Jenkins es un servidor de automatizacion open-source que puedes ejecutar en TU computadora. A diferencia de GitHub Actions (que corre en servidores de GitHub), Jenkins corre 100% localmente en un contenedor Docker. Es como tener tu propio "GitHub Actions privado" en casa.

    Por que usar Jenkins en localhost? GitHub Actions requiere internet y un repositorio en GitHub. Jenkins funciona completamente offline, es ideal para: empresas con codigo privado que no puede salir de su red, desarrollo sin internet, control total del pipeline, y automatizacion de tareas locales (no solo CI/CD).

    Que problema resuelve? Si trabajas en una empresa con politicas de seguridad estrictas que prohiben subir codigo a GitHub, o si necesitas automatizar tareas locales (como generar reportes, hacer backups, o desplegar en servidores internos), Jenkins es la solucion.

    Analogia: Si GitHub Actions es como usar Uber (servicio en la nube, pagas por uso), Jenkins es como tener tu propio auto (lo compras una vez, lo usas cuando quieras, sin depender de nadie).

    // ═══════════════════════════════════════════════════════════════ // Jenkinsfile: Define el pipeline de CI/CD para el Item Service // Este archivo le dice a Jenkins QUE hacer y EN QUE ORDEN // ═══════════════════════════════════════════════════════════════ // pipeline: Bloque principal que define todo el pipeline // Usamos sintaxis declarativa (mas legible que la antigua scripted) pipeline { // agent: Donde se ejecuta el pipeline // 'any' = ejecutar en cualquier agente disponible // En nuestro caso, el agente es el propio contenedor de Jenkins agent any // stages: Secuencia de etapas que se ejecutan en orden // Cada stage es un grupo de pasos relacionados stages { // ═══════════════════════════════════════════════════════ // STAGE 1: Checkout - Descargar el codigo del repositorio // ═══════════════════════════════════════════════════════ stage('Checkout') { steps { // checkout scm: Descarga el codigo del repositorio Git // "scm" = Source Control Management (Git en nuestro caso) // Jenkins usa la configuracion del job para saber de donde clonar checkout scm } } // ═══════════════════════════════════════════════════════ // STAGE 2: Build - Compilar el proyecto Java // ═══════════════════════════════════════════════════════ stage('Build') { steps { // dir: Cambiar al directorio item-service/ // Todos los comandos dentro de este bloque // se ejecutan desde item-service/ dir('item-service') { // sh: Ejecutar comando de shell // mvn clean compile: // clean = borra la carpeta target/ (compilaciones anteriores) // compile = compila el codigo Java (.java -> .class) // Si la compilacion falla, el pipeline se detiene aqui sh 'mvn clean compile' } } } // ═══════════════════════════════════════════════════════ // STAGE 3: Test - Ejecutar tests unitarios // ═══════════════════════════════════════════════════════ stage('Test') { steps { dir('item-service') { // mvn test: Ejecuta todos los tests unitarios // Si algun test falla, el pipeline se marca como "failed" // y Jenkins te notifica (si configuraste notificaciones) sh 'mvn test' } } } } // post: Acciones que se ejecutan DESPUES del pipeline // "always" = ejecutar siempre, sin importar si el pipeline fallo o no post { always { // echo: Imprimir mensaje en la consola de Jenkins // Util para saber cuando termino el pipeline echo 'Pipeline completado. Revisa los resultados arriba.' } success { echo '✓ Pipeline exitoso: Build y tests pasaron correctamente' } failure { echo '✗ Pipeline fallo: Revisa los logs para ver que salio mal' } } }
    🔬 Analisis del Jenkinsfile

    pipeline { }: Bloque principal que define todo el pipeline. Usamos sintaxis declarativa (mas legible que la antigua sintaxis scripted).

    agent any: Indica donde se ejecuta el pipeline. "any" significa "en cualquier agente disponible". En nuestro caso, el agente es el propio contenedor de Jenkins.

    stages { }: Contiene todas las etapas del pipeline. Cada stage es un grupo de pasos relacionados.

    stage('Checkout'): Primera etapa: descargar el codigo del repositorio Git. checkout scm usa la configuracion del job para saber de donde clonar.

    stage('Build'): Segunda etapa: compilar el proyecto Java. dir('item-service') cambia al directorio del microservicio. mvn clean compile limpia compilaciones anteriores y compila el codigo.

    stage('Test'): Tercera etapa: ejecutar tests unitarios. mvn test ejecuta todos los tests. Si alguno falla, el pipeline se detiene.

    post { }: Acciones que se ejecutan despues del pipeline, sin importar si fallo o no. "always" se ejecuta siempre, "success" solo si todo paso, "failure" solo si algo fallo.

    10.5 Frontend en Contenedor con Nginx (Opcional)

    🔬 Por que esta seccion es opcional?

    Situacion actual: Trabajamos directamente con Live Server de VS Code (puerto 5500) para ver el cambio en el frontend. Esto funciona perfectamente para desarrollo porque ofrece hot reload automatico: cuando editas HTML/CSS/JS, el navegador se actualiza instantaneamente sin necesidad de recargar manualmente.

    El "problema": El frontend es el UNICO componente que no corre en un contenedor Docker. PostgreSQL, Item Service, API Gateway y Jenkins todos estan contenedorizados, pero el frontend corre directamente en tu host con Live Server. Esto rompe la consistencia de la arquitectura.

    La solucion: Puedes contenerizar el frontend con Nginx (servidor web ultra-ligero y eficiente). Esto hace que TODO el sistema sea 100% contenedorizado, consistente con la arquitectura de microservicios profesional.

    El trade-off: Pierdes el hot reload automatico. Cuando editas un archivo del frontend, necesitas reconstruir el contenedor para ver los cambios. Por eso esta seccion es opcional: para desarrollo rapido, Live Server es mejor; para produccion o arquitecturas consistentes, Nginx es mejor.

    Analogia: Live Server es como tener tu codigo en una carpeta compartida (rapido, facil, pero informal). Nginx en contenedor es como desplegar tu codigo en un servidor web real (mas formal, pero representa mejor el entorno de produccion).

    Paso 2: Crear configuracion de Nginx

    Crea el archivo frontend/nginx.conf:

    # ═══════════════════════════════════════════════════════════════ # Configuracion de Nginx para el frontend # Este archivo le dice a Nginx cómo manejar las peticiones HTTP # ═══════════════════════════════════════════════════════════════ # server: Define un servidor virtual # Cada "server" block puede manejar diferentes dominios o puertos server { # listen: Puerto en el que Nginx escucha peticiones # 80 = puerto estandar HTTP listen 80; # server_name: Nombre del dominio (para virtual hosts) # "_" = cualquier dominio (comodin) # Como solo tenemos un servidor, usamos "_" para aceptar todo server_name _; # root: Directorio raiz donde Nginx busca los archivos # /usr/share/nginx/html es donde copiamos nuestros archivos en el Dockerfile root /usr/share/nginx/html; # index: Archivo por defecto cuando accedes a la raiz (/) # Si accedes a http://localhost:3000/, Nginx busca index.html index index.html; # location /: Reglas para todas las peticiones location / { # try_files: Intenta servir archivos en este orden: # 1. $uri = la ruta exacta solicitada (ej: /style.css) # 2. $uri/ = la ruta como directorio (ej: /images/) # 3. /index.html = si no existe, sirve index.html (para SPAs) # Esto es importante para aplicaciones de una sola pagina (SPA) try_files $uri $uri/ /index.html; } # location /api/: Reglas para peticiones a la API # Cuando el frontend hace fetch a /api/items, Nginx lo redirige al Gateway location /api/ { # proxy_pass: Redirige la peticion a otro servidor # http://api-gateway:8080 = nombre del servicio en Docker + puerto # Docker DNS resuelve "api-gateway" a la IP del contenedor proxy_pass http://api-gateway:8080; # proxy_set_header: Pasa headers originales al backend # $host = dominio original (ej: localhost:3000) proxy_set_header Host $host; # $remote_addr = IP del cliente (tu navegador) proxy_set_header X-Real-IP $remote_addr; # $http_x_forwarded_for = cadena de proxies (si hay multiples) proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; # $scheme = protocolo (http o https) proxy_set_header X-Forwarded-Proto $scheme; } }
    🔬 Analisis de la configuracion de Nginx

    server { }: Define un servidor virtual. Cada bloque "server" puede manejar diferentes dominios o puertos.

    listen 80: Nginx escucha en el puerto 80 (HTTP estandar). En docker-compose.yml mapearemos este puerto a 3000 del host.

    root /usr/share/nginx/html: Directorio donde Nginx busca los archivos. En el Dockerfile copiamos nuestros archivos del frontend a esta ubicacion.

    location / { }: Reglas para todas las peticiones. try_files intenta servir el archivo exacto, y si no existe, sirve index.html. Esto es importante para aplicaciones de una sola pagina (SPA).

    location /api/ { }: Reglas para peticiones a la API. Cuando el frontend hace fetch('/api/items'), Nginx redirige la peticion al API Gateway usando proxy_pass.

    proxy_pass http://api-gateway:8080: Redirige la peticion al contenedor del API Gateway. Docker DNS resuelve "api-gateway" a la IP del contenedor automaticamente.

    Modulo 11

    Troubleshooting

    Cuando el input no responde: problemas comunes en nuestra arquitectura de microservicios y cómo resolverlos.

    11.1 Conflictos de Puertos

    ⚠ Error: Port already in use
    Si el puerto 8080, 8081 o 5432 esta ocupado:

    Solucion: Identifica el proceso con netstat -ano | findstr :8080 (Windows) o lsof -i :8080 (Linux/Mac) y termimalo, o cambia el puerto en application.yml y docker-compose.yml.

    11.2 Connection Refused

    ⚠ Error: Connection refused to item-service:8081
    El Gateway no puede conectar con el microservicio. Causas posibles:

    1. El contenedor item-service no esta corriendo. Verifica con docker ps.
    2. El microservicio aun no termino de arrancar. Espera 20-30 segundos y revisa los logs: docker logs item-service.
    3. No estan en la misma red Docker. Verifica que ambos servicios esten en crud-network.

    11.3 Errores de Routing del Gateway

    ⚠ Error: 404 Not Found en /api/items
    El Gateway no encuentra la ruta. Verifica:

    1. El predicates en application.yml del Gateway sea Path=/api/items/**.
    2. La uri apunte a http://item-service:8081 (no localhost dentro de Docker).
    3. El microservicio escuche realmente en el puerto 8081.

    11.4 Problemas de CORS

    ⚠ Error: No 'Access-Control-Allow-Origin' header
    El navegador bloquea la peticion por politica CORS. Soluciones:

    1. Verifica que el Gateway tenga globalcors configurado con allowedOrigins: "http://localhost:5500".
    2. Asegurate de que el frontend se sirva desde localhost:5500 (Live Server de VS Code).
    3. NO agregues @CrossOrigin en el microservicio; eso causaria conflictos con el Gateway.
    4. Si usas otro puerto para el frontend, actualiza allowedOrigins en el Gateway.

    11.5 Maven Build Failures

    ⚠ Error: Could not resolve dependencies
    Maven no puede descargar dependencias. Causas:

    1. Sin conexion a internet. Verifica tu conexion.
    2. Repositorio de Maven caido. Espera e intenta de nuevo.
    3. Cache corrupto. Ejecuta mvn clean install -U para forzar actualizacion.

    2 Módulo 6.4 docker compose up -d Arranca los 3 contenedores en orden: postgres → item-service → api-gateway (gracias a depends_on). 30-60s
    3 Módulo 6.4 docker compose ps Verifica que los 3 contenedores estén "Up". 2s
    4 Módulo 10.4 (opcional) docker compose build jenkins
    docker compose up -d    		 Construye y arranca Jenkins para CI/CD. 3-5 min
    5 Módulo 10.5 (opcional) docker compose build frontend
    docker compose up -d    		 Construye y arranca el frontend con Nginx en contenedor. 10-20s
    ⚠️ Notas importantes
    • Comandos 1-5 son obligatorios para que el proyecto funcione. Los comandos 6-7 son opcionales (Módulo 10).
    • Después del comando 2, los contenedores de PostgreSQL e Item Service deben quedar ejecutándose mientras avanzas por el Módulo 6 (luego de completar el Módulo 6.4 se agrega el API Gateway).
    • Si apagas tu computadora entre el Módulo 5.2 y el Módulo 6.4: ejecuta docker compose up -d postgres item-service.
    • Si apagas tu computadora después del Módulo 6.4 (cuando los 3 servicios están listos): ejecuta docker compose up -d para levantar todo.
    • Para detener los contenedores al terminar la sesión: docker compose down (conserva datos).
    • Para empezar de cero (borrar todo, incluyendo BD): docker compose down -v.
    • Si en cualquier momento ves logs de error, revisa el Módulo 11: Troubleshooting.
