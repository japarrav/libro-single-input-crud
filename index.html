🔬 Por que un API Gateway?
  • Enrutamiento: Dirige peticiones al microservicio correcto. /api/items/** va al Item Service.
  • CORS Centralizado: Maneja Cross-Origin en un solo lugar, no en cada microservicio.
  • Seguridad: Puede validar tokens JWT, rate limiting antes de llegar a los servicios.
  • Aislamiento: Los microservicios internos no necesitan estar expuestos al exterior.

0.2 Las 4 Capas del Microservicio (Clean Architecture)

Clean Architecture organiza el codigo en capas concentricas. Cada capa solo depende de la capa inmediatamente inferior, nunca de capas superiores. Esto garantiza que la logica de negocio permanezca aislada de detalles tecnicos como frameworks o bases de datos.

CAPA 4 Controller

Recibe HTTP y devuelve JSON. Convierte DTOs a entidades. Es la unica capa que conoce HTTP.

↓↓ Delega ↓↓
CAPA 3 Service

Logica de negocio, validaciones, transformaciones DTO-Entity. No sabe de HTTP ni de SQL.

↓↓ Delega ↓↓
CAPA 2 Repository

Interfaz con la BD. Spring Data JPA genera SQL automaticamente a partir de nombres de metodos.

↓↓ Mapea ↓↓
CAPA 1 Entity

Representacion Java de la tabla de la base de datos. Define la estructura de los datos.

🔬 Por que separar en capas?

Imagina que quieres cambiar PostgreSQL por MongoDB. Si todo el codigo estuviera mezclado, tendrias que modificar cientos de lineas. Con Clean Architecture, solo cambias la capa Repository y el Entity. Las capas Service y Controller permanecen intactas porque no conocen los detalles de la base de datos.

Otro beneficio: cada capa puede testearse de forma aislada. Puedes probar la logica de negocio (Service) sin necesidad de tener una base de datos corriendo.

Modulo 1

Git: Control de Versiones Profesional

Una maquina del tiempo para tu codigo. Sin Git, no hay ingenieria de software moderna.

1.1 Que es Git y Por Que es Obligatorio?

Git es un sistema de control de versiones distribuido. Cada vez que haces un "commit", tomas una fotografia del estado actual de tu proyecto. Si algo se rompe, puedes volver a cualquier fotografia anterior. Es como tener infinitos "Ctrl+Z" que sobreviven a reinicios, formateos y desastres.

Git es distribuido porque cada desarrollador tiene una copia completa del historial en su maquina. No dependes de un servidor central para trabajar (aunque GitHub/GitLab sirven para sincronizar).

⚠️ Git NO es GitHub

Son herramientas diferentes que trabajan juntas:

  • Git: Software que corre en tu maquina. Gestiona el historial localmente. Puedes usar Git sin GitHub.
  • GitHub/GitLab: Servicios en la nube que almacenan repositorios Git remotamente. Sirven para colaboracion, backup y CI/CD. Son opcionales para desarrollo individual.
💻 CONSOLA - Instalar y Configurar Git
# Verificar instalacion (debe mostrar version 2.40+) git --version # Configurar identidad (obligatorio para commits) # Git necesita saber quien eres para registrar los cambios git config --global user.name "Tu Nombre" git config --global user.email "tu@email.com" # Verificar configuracion git config --global --list # Establecer rama principal como "main" (estandar moderno) git config --global init.defaultBranch main

La bandera --global aplica la configuracion a todos tus repositorios. Sin ella, solo aplica al repositorio actual.

1.2 Inicializar Repositorio y .gitignore

Un repositorio es la carpeta donde Git rastrea cambios. Inicializarlo crea una carpeta oculta .git/ que contiene todo el historial.

💻 CONSOLA - git init
# Ir a la carpeta del proyecto cd estudio-docker-microserv # Inicializar repositorio (crea carpeta .git/ oculta) git init # Ver estado (todo esta "untracked" = sin rastrear) git status # Crear .gitignore ANTES del primer commit # (ver codigo abajo) # Anadir todos los archivos al "staging area" git add . # Primer commit (fotografia inicial del proyecto) git commit -m "Initial commit: estructura base del proyecto"

Conceptos clave:

  • git add . marca todos los archivos para el proximo commit.
  • git commit toma la fotografia. El mensaje (-m) describe que cambios contiene.
  • git status muestra que archivos han cambiado desde el ultimo commit.

Crea .gitignore en la raiz. Este archivo le dice a Git que ignore ciertos archivos (credenciales, compilados, configuraciones locales):

# ═══════════════════════════════════════════════════════════════ # ARCHIVOS QUE GIT DEBE IGNORAR # ═══════════════════════════════════════════════════════════════ # Variables de entorno con credenciales (NUNCA subir secretos) .env .env.local .env.*.local # IDE y editores (configuraciones personales) .vscode/ .idea/ *.iml # Archivos de sistema .DS_Store Thumbs.db # Java compilado (Maven lo compila dentro del contenedor) target/ *.class *.jar *.war # Logs *.log logs/ # Node (si hubiera frontend Node) node_modules/ # Docker (no incluir en build context) .docker/
⚠️ Por que ignorar .env?

El archivo .env contiene credenciales (usuario, contraseña de PostgreSQL). Si lo subes a Git (especialmente a GitHub publico), cualquiera podra ver tus secretos. Por eso .gitignore lo excluye. En su lugar, sube un archivo .env.example con valores de ejemplo (sin credenciales reales) para que otros desarrolladores sepan que variables necesitan.

1.3 Commits y Ramas: Flujo de Trabajo

Las ramas (branches) te permiten trabajar en funcionalidades aisladas sin afectar la rama principal. Cuando terminas, "mergeas" (fusionas) la rama de vuelta a main.

Flujo de Ramas

main:       A---B-----------E---F (merge)
                 \         /
feature:          C---D---

A, B = commits iniciales. C, D = trabajo en feature. E = merge. F = siguiente commit en main.

💻 CONSOLA - Flujo de Trabajo con Ramas
# Renombrar rama principal a main (estandar moderno) git branch -M main # Crear rama para nueva funcionalidad y cambiar a ella # "-b" crea la rama, "checkout" cambia a ella git checkout -b feature/item-service # Trabajar, hacer cambios en archivos... # Cuando termines una unidad logica de trabajo: git add . git commit -m "Add Item entity and repository" # Otro cambio (por ejemplo, agregar Controller): git add . git commit -m "Add ItemController with 6 endpoints" # Ver historial de commits (grafico de ramas) git log --oneline --graph # Volver a rama main git checkout main # Fusionar la rama feature a main git merge feature/item-service # Ver todas las ramas (la rama feature sigue existiendo) git branch -a # Eliminar rama feature (ya no la necesitamos) git branch -d feature/item-service
⚠️ Buenas Practicas de Commits
  • Commits atomicos: Cada commit = un cambio logico unico. No mezcles "añadir entity" con "configurar docker".
  • Mensajes en imperativo: "Add feature", "Fix bug", "Update config". No "Added", "Fixed", "Updated".
  • Nunca subas .env: Contiene credenciales. Siempre en .gitignore.
  • Frecuencia: Commit despues de completar una tarea pequena. Mejor muchos commits pequeños que uno gigante.

1.4 Clonar un Repositorio

Si el proyecto ya existe en GitHub (o tu quieres trabajar en otra maquina), usas git clone para descargar una copia completa con todo el historial.

💻 CONSOLA - Clonar Repositorio
# Clonar repositorio publico de GitHub git clone https://github.com/usuario/single-input-crud.git # Esto crea una carpeta "single-input-crud/" con todo el contenido # e inicializa Git automaticamente (no necesitas git init) # Entrar a la carpeta cd single-input-crud # Ver historial de commits git log --oneline # Ver ramas disponibles git branch -a # Si quieres trabajar en una rama remota: git checkout -b feature/nueva-funcionalidad origin/feature/nueva-funcionalidad # Despues de hacer cambios, subirlos al repositorio remoto: git add . git commit -m "Add new feature" git push origin feature/nueva-funcionalidad

Conceptos clave:

  • clone descarga TODO: archivos + historial completo + ramas.
  • origin es el nombre por defecto del repositorio remoto (de donde clonaste).
  • push sube tus commits locales al repositorio remoto.
  • pull es lo opuesto: descarga cambios del remoto a tu local.

Modulo 2

Docker: Contenedores, PostgreSQL y Redes

Construir los cimientos con aislamiento completo. Cada servicio vive en su propio universo.

2.1 Contenedor vs Maquina Virtual

Un contenedor es un proceso del sistema operativo aislado. No es una maquina virtual completa (no tiene su propio kernel), pero tiene su propio sistema de archivos, tabla de procesos, red y espacio IPC. Comparte el kernel del host con otros contenedores.

Maquina Virtual (VM)

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|
| Guest OS | | Guest OS |  <- OS completo por VM
+----------+-+----------+
|     Hypervisor       |  <- Capa pesada
+----------------------+
|      Host OS         |
+----------------------+

Contenedor Docker

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|  <- Solo lo necesario
+----------+-+----------+
|    Docker Engine     |  <- Capa ligera
+----------------------+
|   Host OS Kernel     |  <- Compartido
+----------------------+

La consecuencia practica es brutal: un contenedor arranca en menos de 1 segundo (una VM tarda 30-60s), consume ~50MB de RAM (una VM necesita 1-2GB), y la imagen de PostgreSQL Alpine pesa solo ~80MB (una VM con Linux + PostgreSQL serian varios GB).

2.2 Docker Compose: Los 3 Servicios

En vez de usar comandos docker run con muchas banderas, usaremos Docker Compose para definir todos los servicios en un archivo YAML legible y versionable.

Primero, crea el archivo .env en la raiz del proyecto. Este archivo contiene las credenciales y NO se sube a Git:

# ═══════════════════════════════════════════════════════════════ # VARIABLES DE ENTORNO (NO subir a Git - esta en .gitignore) # ═══════════════════════════════════════════════════════════════ # PostgreSQL POSTGRES_USER=admin POSTGRES_PASSWORD=secret123 POSTGRES_DB=crud_db

Ahora crea docker-compose.yml en la raiz:

services: # ═══════════════════════════════════════════════════════ # SERVICIO 1: Base de Datos PostgreSQL 16 (Alpine) # ═══════════════════════════════════════════════════════ postgres: image: postgres:16-alpine container_name: crud-postgres restart: unless-stopped environment: POSTGRES_USER: ${POSTGRES_USER} POSTGRES_PASSWORD: ${POSTGRES_PASSWORD} POSTGRES_DB: ${POSTGRES_DB} ports: - "5432:5432" volumes: - crud_pgdata:/var/lib/postgresql/data healthcheck: test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER} -d ${POSTGRES_DB}"] interval: 10s timeout: 5s retries: 5 start_period: 30s networks: - crud-network # ═══════════════════════════════════════════════════════ # SERVICIO 2: Item Service (Spring Boot Microservice) # ═══════════════════════════════════════════════════════ item-service: build: context: ./item-service dockerfile: Dockerfile container_name: crud-item-service restart: unless-stopped depends_on: postgres: condition: service_healthy environment: SPRING_DATASOURCE_URL: jdbc:postgresql://crud-postgres:5432/crud_db SPRING_DATASOURCE_USERNAME: admin SPRING_DATASOURCE_PASSWORD: secret123 ports: - "8081:8081" - "5005:5005" volumes: - ./item-service/src:/app/src - ./item-service/pom.xml:/app/pom.xml - maven_cache:/root/.m2 networks: - crud-network # ═══════════════════════════════════════════════════════ # SERVICIO 3: API Gateway (Spring Cloud Gateway) # ═══════════════════════════════════════════════════════ api-gateway: build: context: ./api-gateway dockerfile: Dockerfile container_name: crud-api-gateway restart: unless-stopped depends_on: - item-service ports: - "8080:8080" volumes: - ./api-gateway/src:/app/src - ./api-gateway/pom.xml:/app/pom.xml - gateway_maven_cache:/root/.m2 networks: - crud-network volumes: crud_pgdata: driver: local maven_cache: driver: local gateway_maven_cache: driver: local networks: crud-network: driver: bridge

2.3 Las 3 Reglas de Oro de las IPs en Docker

Este es el concepto mas importante del libro. Si lo entiendes, todo lo demas sera natural. Si lo ignoras, pasaras horas debuggeando errores de conexion.

⚠️ REGLAS FUNDAMENTALES

Regla 1: "localhost" es egosta

Dentro de un contenedor, localhost = el mismo contenedor. Si Item Service dice localhost:5432, busca PostgreSQL dentro de SI MISMO (y falla). Nunca uses localhost entre contenedores.

Regla 2: Los contenedores se llaman por nombre

Las IPs (172.17.0.x) son efimeras. El nombre (crud-postgres) es estable. Docker DNS resuelve nombres a IPs. Por eso usamos host: crud-postgres en application.yml.

Regla 3: El Host ve contenedores por puertos mapeados

Desde tu navegador, accedes con localhost:PUERTO_HOST. El mapeo -p 8080:8080 crea un tunel entre tu maquina y el contenedor.

2.4 Volumenes: Persistencia de Datos

Sin volumen, si eliminas el contenedor, todos los datos se pierden. Con volumen, los datos sobreviven a la destruccion del contenedor.

💻 CONSOLA - Comandos de Volumenes
# Levantar todo docker compose up -d # Ver estado docker compose ps # Ver volumenes docker volume ls # Ver logs en tiempo real docker compose logs -f # Detener (conserva volumenes/datos) docker compose down # Detener y BORRAR TODO (incluye datos!) docker compose down -v
⚠️ Cuidado con docker compose down -v

El flag -v elimina los volumenes nombrados. Esto significa que todos los datos de PostgreSQL se borran permanentemente. Usalo solo cuando quieras empezar de cero.

Modulo 3

Desarrollo en Vivo: Dockerfiles y Bind Mounts

Cada Ctrl+S se refleja automaticamente en el contenedor. Sin reinicios manuales.

3.1 Dockerfile del Item Service

Un Dockerfile es la "receta" que Docker usa para construir una imagen. Cada instruccion (FROM, RUN, COPY, etc.) crea una "capa" en la imagen. Docker cachea estas capas: si no cambias una instruccion, no re-ejecuta esa capa.

Crea item-service/Dockerfile:

# Imagen base: JDK 21 + Maven 3.9 sobre Alpine Linux # Alpine es una distribucion ultra-ligera (~5MB) FROM maven:3.9-eclipse-temurin-21-alpine # Instalar bash y curl (Alpine trae solo sh por defecto) # --no-cache evita guardar el indice de paquetes (ahorra ~5MB) RUN apk add --no-cache bash curl # Establecer directorio de trabajo dentro del contenedor # Todos los comandos siguientes se ejecutan desde /app WORKDIR /app # Copiar SOLO pom.xml primero (optimizacion de cache) # Si pom.xml no cambia, Docker usa la capa cacheada COPY pom.xml ./pom.xml # Descargar TODAS las dependencias Maven # -B = batch mode (sin colores, mejor para logs) # Esta capa se cachea: si no cambias pom.xml, no se re-ejecuta RUN mvn dependency:go-offline -B # Variable de entorno: perfil activo de Spring Boot ENV SPRING_PROFILES_ACTIVE=dev # Documentacion: el contenedor escucha en estos puertos # El mapeo real lo hace docker-compose.yml con -p EXPOSE 8081 EXPOSE 5005 # Comando para arrancar Spring Boot # fork=true activa los dos classloaders de DevTools # jvmArguments habilita debug remoto en puerto 5005 CMD ["mvn", "spring-boot:run", \ "-Dspring-boot.run.fork=true", \ "-Dspring-boot.run.jvmArguments=-Xdebug -Xrunjdwp:transport=dt_socket,server=y,suspend=n,address=*:5005"]
🔬 Desglose del Dockerfile
DirectivaExplicacion

COPY pom.xml ./pom.xmlCopia SOLO pom.xml primero. Docker cachea la descarga de dependencias. Si no cambias pom.xml, no se re-ejecuta.

Crea api-gateway/Dockerfile:

FROM maven:3.9-eclipse-temurin-21-alpine RUN apk add --no-cache bash curl WORKDIR /app COPY pom.xml ./pom.xml RUN mvn dependency:go-offline -B EXPOSE 8080 CMD ["mvn", "spring-boot:run"]

3.2 Construir y Arrancar Todo

💻 CONSOLA - Construir y Arrancar
# Construir imagenes (primera vez: 5-10 min) docker compose build # Arrancar los 3 servicios en segundo plano docker compose up -d # Ver estado (los 3 deben estar Up) docker compose ps # Ver logs del Item Service docker compose logs -f item-service # Espera: "Started ItemServiceApplication in X.XXX seconds" # Ver logs del API Gateway docker compose logs -f api-gateway # Espera: "Started ApiGatewayApplication in X.XXX seconds" # Ver logs de todos los servicios docker compose logs -f
⚠️ Primera construccion es lenta

La primera vez que ejecutas docker compose build, Docker descarga las imagenes base y todas las dependencias Maven. Esto tarda 5-10 minutos. Las siguientes veces sera mucho mas rapido gracias al cache de capas.

3.3 Archivo .dockerignore

Cuando ejecutas docker compose build, Docker envia todo el contenido de la carpeta actual (el "build context") al daemon. Esto incluye archivos que no necesitas: .git, node_modules, archivos temporales, etc. Para evitar esto, crea un archivo .dockerignore en la raiz del proyecto.

# Control de versiones .git .gitignore # IDE y editores .vscode .idea *.iml # Archivos de sistema .DS_Store Thumbs.db # Docker (no incluir en build context) Dockerfile docker-compose.yml .dockerignore # Documentacion (no necesaria en la imagen) *.md README* # Variables de entorno (NUNCA subir secretos) .env # Java compilado (Maven lo compila dentro del contenedor) target/ *.class *.jar # Node (si hubiera frontend Node) node_modules/
🔬 Por que es importante .dockerignore?
BeneficioExplicacion
Velocidad de buildSin .git (que puede ser cientos de MB), el build context se envia en segundos en vez de minutos.
Tamano de imagenArchivos innecesarios no entran en la imagen, reduciendo su tamano final.
SeguridadArchivos .env con credenciales no se filtran accidentalmente en la imagen.
Cache de capasMenos archivos = menos cambios detectados = mejor aprovechamiento del cache de Docker.

package com.singleinput.service; import com.singleinput.domain.Item; import com.singleinput.dto.ItemDTO; import com.singleinput.exception.ItemNotFoundException; import com.singleinput.persistence.ItemRepository; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import java.util.List; import java.util.stream.Collectors; @Service public class ItemServiceImpl implements ItemService { private final ItemRepository repo; public ItemServiceImpl(ItemRepository repo) { this.repo = repo; } @Override @Transactional(readOnly = true) public List<ItemDTO> findAll() { return repo.findAll().stream().map(this::toDTO).collect(Collectors.toList()); } @Override @Transactional(readOnly = true) public ItemDTO findById(Long id) { return toDTO(repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id))); } @Override @Transactional public ItemDTO create(ItemDTO dto) { Item e = new Item(); e.setTexto(dto.getTexto()); return toDTO(repo.save(e)); } @Override @Transactional public ItemDTO update(Long id, ItemDTO dto) { Item ex = repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id)); ex.setTexto(dto.getTexto()); return toDTO(repo.save(ex)); } @Override @Transactional public void delete(Long id) { if (!repo.existsById(id)) throw new ItemNotFoundException(id); repo.deleteById(id); } @Override @Transactional(readOnly = true) public List<ItemDTO> search(String texto) { return repo.findByTextoContainingIgnoreCase(texto).stream().map(this::toDTO).collect(Collectors.toList()); } private ItemDTO toDTO(Item i) { ItemDTO d = new ItemDTO(); d.setId(i.getId()); d.setTexto(i.getTexto()); return d; } }

Modulo 5

API Gateway con Spring Cloud Gateway

El punto de entrada unico para todas las peticiones del cliente.

5.1 Que es un API Gateway?

Un API Gateway es un servidor que actua como puerta de entrada unica a todos los microservicios. El cliente (navegador) no sabe cuantos microservicios existen ni en que puertos corren; solo conoce la direccion del Gateway.

ⓘ Responsabilidades del Gateway
Enrutamiento: Redirige /api/items/** al microservicio item-service:8081.
CORS: Maneja los headers de Cross-Origin para permitir peticiones desde el frontend.
Seguridad: Punto centralizado para autenticacion, rate limiting, logging.
WebFlux: Spring Cloud Gateway usa programacion reactiva (NO MVC), lo que lo hace muy eficiente.

Modulo 8

JWT: Autenticacion Opcional (Sin Base de Datos)

Implementar autenticacion con JWT usando credenciales en un archivo de configuracion.

8.1 Que es JWT?

JWT (JSON Web Token) es un estandar para crear tokens de autenticacion. Un JWT es una cadena de texto firmada que contiene informacion del usuario (claims). El servidor verifica la firma sin necesidad de consultar una base de datos.

ⓘ Estructura de un JWT
Un JWT tiene 3 partes separadas por puntos:
eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJhZG1pbiJ9.SflKxwRJSMeKKF2QT4fwpM

Header: Algoritmo de firma (HS256).
Payload: Claims (datos del usuario: sub, exp, etc.).
Signature: Firma criptografica para verificar integridad.

8.2 Configuracion sin Base de Datos

Para este ejemplo, usaremos credenciales hardcodeadas en application.yml del Gateway. En produccion, usarias un servicio de autenticacion externo o una base de datos.

<!-- Agregar esta dependencia al pom.xml del Gateway --> <dependency> <groupId>io.jsonwebtoken</groupId> <artifactId>jjwt-api</artifactId> <version>0.12.6</version> </dependency> <dependency> <groupId>io.jsonwebtoken</groupId> <artifactId>jjwt-impl</artifactId> <version>0.12.6</version> <scope>runtime</scope> </dependency> <dependency> <groupId>io.jsonwebtoken</groupId> <artifactId>jjwt-jackson</artifactId> <version>0.12.6</version> <scope>runtime</scope> </dependency>

Agregar credenciales y clave secreta en api-gateway/src/main/resources/application.yml:

security: jwt: secret: mi-clave-secreta-muy-larga-para-jwt-que-debe-tener-al-menos-256-bits expiration: 3600000 users: - username: admin password: admin123 role: ADMIN - username: user password: user123 role: USER

Crear clase para generar y validar JWT en api-gateway/src/main/java/com/gateway/security/JwtUtil.java:

package com.gateway.security; import io.jsonwebtoken.*; import org.springframework.beans.factory.annotation.Value; import org.springframework.stereotype.Component; import java.util.Date; @Component public class JwtUtil { @Value("") private String secret; @Value("") private long expiration; public String generateToken(String username, String role) { return Jwts.builder() .setSubject(username) .claim("role", role) .setIssuedAt(new Date()) .setExpiration(new Date(System.currentTimeMillis() + expiration)) .signWith(SignatureAlgorithm.HS256, secret) .compact(); } public String validateToken(String token) { try { Claims claims = Jwts.parser() .setSigningKey(secret) .parseClaimsJws(token) .getBody(); return claims.getSubject(); } catch (Exception e) { return null; } } }

Modulo 9

CI/CD: Integracion y Despliegue Continuo

Automatizar build, test y deploy con GitHub Actions.

9.1 Que es CI/CD?

CI (Integracion Continua): Cada vez que haces push a Git, se ejecutan automaticamente: build, tests, analisis de codigo. Si algo falla, te notifican inmediatamente.

CD (Despliegue Continuo): Si el CI pasa, el codigo se despliega automaticamente a staging o produccion. Sin intervencion manual.

ⓘ Beneficios de CI/CD
Deteccion temprana de errores: Los tests se ejecutan en cada push.
Despliegues rapidos: De horas/manuales a minutos/automaticos.
Consistencia: El mismo proceso se ejecuta siempre, sin errores humanos.
Confianza: Si los tests pasan, puedes desplegar con seguridad.

9.2 GitHub Actions

GitHub Actions es el sistema de CI/CD de GitHub. Defines "workflows" en archivos YAML dentro de .github/workflows/. Cada workflow se dispara automaticamente segun eventos (push, pull request, etc.).

9.3 Pipeline para Item Service

Crea .github/workflows/ci-item-service.yml:

name: CI Item Service on: push: branches: [main] paths: - 'item-service/**' pull_request: branches: [main] paths: - 'item-service/**' jobs: build-and-test: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Set up JDK 21 uses: actions/setup-java@v4 with: java-version: '21' distribution: 'temurin' cache: 'maven' - name: Build with Maven working-directory: ./item-service run: mvn clean package -DskipTests - name: Run tests working-directory: ./item-service run: mvn test - name: Upload test results if: always() uses: actions/upload-artifact@v4 with: name: test-results path: item-service/target/surefire-reports/

Modulo 10

Troubleshooting

Problemas comunes y como resolverlos.

10.1 Conflictos de Puertos

⚠ Error: Port already in use
Si el puerto 8080, 8081 o 5432 esta ocupado:

Solucion: Identifica el proceso con netstat -ano | findstr :8080 (Windows) o lsof -i :8080 (Linux/Mac) y termimalo, o cambia el puerto en application.yml y docker-compose.yml.

10.2 Connection Refused

⚠ Error: Connection refused to item-service:8081
El Gateway no puede conectar con el microservicio. Causas posibles:

1. El contenedor item-service no esta corriendo. Verifica con docker ps.
2. El microservicio aun no termino de arrancar. Espera 20-30 segundos y revisa los logs: docker logs item-service.
3. No estan en la misma red Docker. Verifica que ambos servicios esten en crud-network.

10.3 Errores de Routing del Gateway

⚠ Error: 404 Not Found en /api/items
El Gateway no encuentra la ruta. Verifica:

1. El predicates en application.yml del Gateway sea Path=/api/items/**.
2. La uri apunte a http://item-service:8081 (no localhost dentro de Docker).
3. El microservicio escuche realmente en el puerto 8081.

10.4 Problemas de CORS

⚠ Error: No 'Access-Control-Allow-Origin' header
El navegador bloquea la peticion por politica CORS. Soluciones:

1. Verifica que el Gateway tenga globalcors configurado con allowedOrigins: "http://localhost:5500".
2. Asegurate de que el frontend se sirva desde localhost:5500 (Live Server de VS Code).
3. NO agregues @CrossOrigin en el microservicio; eso causaria conflictos con el Gateway.
4. Si usas otro puerto para el frontend, actualiza allowedOrigins en el Gateway.

10.5 Maven Build Failures

⚠ Error: Could not resolve dependencies
Maven no puede descargar dependencias. Causas:

1. Sin conexion a internet. Verifica tu conexion.
2. Repositorio de Maven caido. Espera e intenta de nuevo.
3. Cache corrupto. Ejecuta mvn clean install -U para forzar actualizacion.
