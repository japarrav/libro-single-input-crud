🔬 Por que un API Gateway?
  • Enrutamiento: Dirige peticiones al microservicio correcto. /api/items/** va al Item Service.
  • CORS Centralizado: Maneja Cross-Origin en un solo lugar, no en cada microservicio.
  • Seguridad: Puede validar tokens JWT, rate limiting antes de llegar a los servicios.
  • Aislamiento: Los microservicios internos no necesitan estar expuestos al exterior.

Modulo 1

Git: Control de Versiones Profesional

Una maquina del tiempo para tu codigo.

1.1 Que es Git?

Git es un sistema de control de versiones distribuido. Cada vez que haces un "commit", tomas una fotografia del estado actual de tu proyecto. Si algo se rompe, puedes volver a cualquier fotografia anterior.

⚠️ Git NO es GitHub
  • Git: Software que corre en tu maquina. Gestiona el historial localmente.
  • GitHub/GitLab: Servicios en la nube que almacenan repositorios Git. Son opcionales.
💻 CONSOLA - Instalar y Configurar Git
# Verificar instalacion git --version # Salida: git version 2.40.0 (o superior) # Configurar identidad (obligatorio para commits) git config --global user.name "Tu Nombre" git config --global user.email "tu@email.com" # Verificar git config --global --list

1.3 Commits y Ramas

💻 CONSOLA - Flujo de Trabajo con Ramas
# Renombrar rama principal a main git branch -M main # Crear rama para nueva funcionalidad git checkout -b feature/item-service # Trabajar, hacer cambios... git add . git commit -m "Add Item entity and repository" # Otro cambio git add . git commit -m "Add API Gateway configuration" # Ver historial git log --oneline --graph # Volver a main y mergear git checkout main git merge feature/item-service # Ver todas las ramas git branch -a
⚠️ Buenas Practicas
  • Commits atomicos: Cada commit = un cambio logico unico.
  • Mensajes en imperativo: "Add feature", "Fix bug", no "Added", "Fixed".
  • Nunca subas .env: Contiene credenciales.

Modulo 2

Docker: Contenedores, PostgreSQL y Redes

Construir los cimientos con aislamiento completo.

2.1 Contenedor vs Maquina Virtual

Un contenedor es un proceso aislado del SO. No tiene su propio kernel (comparte el del host), pero tiene su propio sistema de archivos, red y procesos.

Maquina Virtual (VM)

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|
| Guest OS | | Guest OS |  <- OS completo
+----------+-+----------+
|     Hypervisor       |  <- Pesado
+----------------------+
|      Host OS         |
+----------------------+

Contenedor Docker

+----------+ +----------+
| App      | | App      |
| Bins/Libs| | Bins/Libs|  <- Solo necesario
+----------+-+----------+
|    Docker Engine     |  <- Ligero
+----------------------+
|   Host OS Kernel     |  <- Compartido
+----------------------+

Contenedor: arranca en <1 segundo, consume ~50MB RAM. PostgreSQL Alpine pesa solo ~80MB.

2.3 Las 3 Reglas de Oro de las IPs en Docker

⚠️ REGLAS FUNDAMENTALES

Regla 1: "localhost" es egosta

Dentro de un contenedor, localhost = el mismo contenedor. Si Item Service dice localhost:5432, busca PostgreSQL dentro de SI MISMO (y falla).

Regla 2: Los contenedores se llaman por nombre

Las IPs (172.17.0.x) son efimeras. El nombre (crud-postgres) es estable. Docker DNS resuelve nombres a IPs.

Regla 3: El Host ve contenedores por puertos mapeados

Desde tu navegador, accedes con localhost:PUERTO_HOST. El mapeo -p 8080:8080 crea un tunel.

2.4 Volumenes: Persistencia de Datos

Sin volumen = datos efimeros. Con volumen = datos sobreviven a la destruccion del contenedor.

💻 CONSOLA - Comandos de Volumenes
# Levantar todo docker compose up -d # Ver estado docker compose ps # Ver volumenes docker volume ls # Ver logs en tiempo real docker compose logs -f # Detener (conserva volumenes/datos) docker compose down # Detener y BORRAR TODO (incluye datos!) docker compose down -v

service/ItemServiceImpl.java (implementacion):

package com.singleinput.service; import com.singleinput.domain.Item; import com.singleinput.dto.ItemDTO; import com.singleinput.exception.ItemNotFoundException; import com.singleinput.persistence.ItemRepository; import org.springframework.stereotype.Service; import org.springframework.transaction.annotation.Transactional; import java.util.List; import java.util.stream.Collectors; @Service public class ItemServiceImpl implements ItemService { private final ItemRepository repo; public ItemServiceImpl(ItemRepository repo) { this.repo = repo; } @Override @Transactional(readOnly = true) public List<ItemDTO> findAll() { return repo.findAll().stream().map(this::toDTO).collect(Collectors.toList()); } @Override @Transactional(readOnly = true) public ItemDTO findById(Long id) { return toDTO(repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id))); } @Override @Transactional public ItemDTO create(ItemDTO dto) { Item e = new Item(); e.setTexto(dto.getTexto()); return toDTO(repo.save(e)); } @Override @Transactional public ItemDTO update(Long id, ItemDTO dto) { Item ex = repo.findById(id).orElseThrow(() -> new ItemNotFoundException(id)); ex.setTexto(dto.getTexto()); return toDTO(repo.save(ex)); } @Override @Transactional public void delete(Long id) { if (!repo.existsById(id)) throw new ItemNotFoundException(id); repo.deleteById(id); } @Override @Transactional(readOnly = true) public List<ItemDTO> search(String texto) { return repo.findByTextoContainingIgnoreCase(texto).stream().map(this::toDTO).collect(Collectors.toList()); } private ItemDTO toDTO(Item i) { ItemDTO d = new ItemDTO(); d.setId(i.getId()); d.setTexto(i.getTexto()); return d; } }

controller/ItemController.java:

package com.singleinput.controller; import com.singleinput.dto.ItemDTO; import com.singleinput.service.ItemService; import jakarta.validation.Valid; import org.springframework.http.ResponseEntity; import org.springframework.web.bind.annotation.*; import java.util.List; @RestController @RequestMapping("/api/items") public class ItemController { private final ItemService service; public ItemController(ItemService service) { this.service = service; } @GetMapping public ResponseEntity<List<ItemDTO>> findAll() { return ResponseEntity.ok(service.findAll()); } @GetMapping("/{id}") public ResponseEntity<ItemDTO> findById(@PathVariable Long id) { return ResponseEntity.ok(service.findById(id)); } @GetMapping("/search") public ResponseEntity<List<ItemDTO>> search(@RequestParam String texto) { return ResponseEntity.ok(service.search(texto)); } @PostMapping public ResponseEntity<ItemDTO> create(@Valid @RequestBody ItemDTO dto) { return ResponseEntity.ok(service.create(dto)); } @PutMapping("/{id}") public ResponseEntity<ItemDTO> update(@PathVariable Long id, @Valid @RequestBody ItemDTO dto) { return ResponseEntity.ok(service.update(id, dto)); } @DeleteMapping("/{id}") public ResponseEntity<Void> delete(@PathVariable Long id) { service.delete(id); return ResponseEntity.noContent().build(); } }

Modulo 5

API Gateway con Spring Cloud Gateway

El punto de entrada unico para todas las peticiones del cliente.

5.1 Que es un API Gateway?

Un API Gateway es un servidor que actua como puerta de entrada unica a todos los microservicios. El cliente (navegador) no sabe cuantos microservicios existen ni en que puertos corren; solo conoce la direccion del Gateway.

ⓘ Responsabilidades del Gateway
Enrutamiento: Redirige /api/items/** al microservicio item-service:8081.
CORS: Maneja los headers de Cross-Origin para permitir peticiones desde el frontend.
Seguridad: Punto centralizado para autenticacion, rate limiting, logging.
WebFlux: Spring Cloud Gateway usa programacion reactiva (NO MVC), lo que lo hace muy eficiente.

ⓘ Flujo de una Peticion
1. Postman envia POST http://localhost:8080/api/items
2. El Gateway recibe la peticion y verifica que la ruta coincida con /api/items/**
3. El Gateway reenvia la peticion a http://item-service:8081/api/items (resolucion DNS de Docker)
4. El microservicio procesa, guarda en PostgreSQL y devuelve la respuesta
5. El Gateway agrega headers CORS y devuelve la respuesta a Postman

6.2 Variables Dinamicas en Postman

Para no repetir URLs, crea una variable de entorno en Postman:

Variable: baseUrl Valor: http://localhost:8080/api/items Usar en requests: {{baseUrl}} -- GET all, POST {{baseUrl}}/1 -- GET by ID, PUT, DELETE {{baseUrl}}/search?texto=Hola -- Search

7.3 app.js

frontend/app.js - Fetch API con async/await, renderizado seguro con textContent (previene XSS):

const API = 'http://localhost:8080/api/items'; const input = document.getElementById('itemInput'); const hiddenId = document.getElementById('itemId'); const list = document.getElementById('itemList'); document.getElementById('btnCreate').addEventListener('click', createItem); document.getElementById('btnUpdate').addEventListener('click', updateItem); document.getElementById('btnDelete').addEventListener('click', deleteItem); document.getElementById('btnSearch').addEventListener('click', searchItems); async function loadAll() { const res = await fetch(API); const data = await res.json(); render(data); } async function createItem() { const texto = input.value.trim(); if (!texto) return; await fetch(API, { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ texto }) }); input.value = ''; loadAll(); } async function updateItem() { const id = hiddenId.value; const texto = input.value.trim(); if (!id || !texto) return; await fetch(`${API}/${id}`, { method: 'PUT', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ texto }) }); input.value = ''; hiddenId.value = ''; loadAll(); } async function deleteItem() { const id = hiddenId.value; if (!id) return; await fetch(`${API}/${id}`, { method: 'DELETE' }); input.value = ''; hiddenId.value = ''; loadAll(); } async function searchItems() { const texto = input.value.trim(); if (!texto) { loadAll(); return; } const res = await fetch(`${API}/search?texto=${encodeURIComponent(texto)}`); const data = await res.json(); render(data); } function render(items) { list.innerHTML = ''; items.forEach(item => { const li = document.createElement('li'); const span = document.createElement('span'); span.textContent = item.texto; const idSpan = document.createElement('span'); idSpan.className = 'item-id'; idSpan.textContent = '#' + item.id; li.appendChild(span); li.appendChild(idSpan); li.addEventListener('click', () => { input.value = item.texto; ⚠ Seguridad XSS
Nunca uses innerHTML para renderizar datos del servidor. Usamos textContent para que cualquier intento de inyectar HTML/JavaScript sea tratado como texto plano. El campo input type="hidden" guarda el ID del item seleccionado para las operaciones de Update y Delete.

Modulo 8

Troubleshooting

Problemas comunes y como resolverlos.

8.1 Conflictos de Puertos

⚠ Error: Port already in use
Si el puerto 8080, 8081 o 5432 esta ocupado:

Solucion: Identifica el proceso con netstat -ano | findstr :8080 (Windows) o lsof -i :8080 (Linux/Mac) y termimalo, o cambia el puerto en application.yml y docker-compose.yml.

8.2 Connection Refused

⚠ Error: Connection refused to item-service:8081
El Gateway no puede conectar con el microservicio. Causas posibles:

1. El contenedor item-service no esta corriendo. Verifica con docker ps.
2. El microservicio aun no termino de arrancar. Espera 20-30 segundos y revisa los logs: docker logs item-service.
3. No estan en la misma red Docker. Verifica que ambos servicios esten en single-input-network.

8.3 Errores de Routing del Gateway

⚠ Error: 404 Not Found en /api/items
El Gateway no encuentra la ruta. Verifica:

1. El predicates en application.yml del Gateway sea Path=/api/items/**.
2. La uri apunte a http://item-service:8081 (no localhost dentro de Docker).
3. El microservicio escuche realmente en el puerto 8081.

8.4 Problemas de CORS

⚠ Error: No 'Access-Control-Allow-Origin' header
El navegador bloquea la peticion por politica CORS. Soluciones:

1. Verifica que el Gateway tenga globalcors configurado con allowedOrigins: "http://localhost:5500".
2. Asegurate de que el frontend se sirva desde localhost:5500 (Live Server de VS Code).
3. NO agregues @CrossOrigin en el microservicio; eso causaria conflictos con el Gateway.
4. Si usas otro puerto para el frontend, actualiza allowedOrigins en el Gateway.

Modulo 9

Siguientes Pasos y Glosario

Hacia donde llevar esta arquitectura y conceptos clave.

9.1 Siguientes Pasos

ⓘ Extensiones Recomendadas
1. Service Discovery: Agregar Eureka o Consul para que los servicios se descubran automaticamente sin hardcodear URLs.
2. Circuit Breaker: Implementar Resilience4j para manejar fallos entre servicios gracefully.
3. Config Server: Centralizar configuraciones con Spring Cloud Config Server.
4. Autenticacion: Agregar Spring Security + JWT en el Gateway.
5. Logging Centralizado: Usar ELK Stack (Elasticsearch, Logstash, Kibana) o Loki.
6. Monitoring: Integrar Prometheus + Grafana para metricas en tiempo real.
7. CI/CD: Automatizar build, test y deploy con GitHub Actions o Jenkins.
8. Kubernetes: Migrar de Docker Compose a Kubernetes para orquestacion en produccion.
